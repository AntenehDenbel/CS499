What the Review Covers

The code review is a roughly thirty minute screencast directed at a peer reviewer or engineering manager audience. It walks the existing application through three layers: a brief demonstration of the user-facing behavior on the emulator, a tour of the project structure as it stood at the end of CS 360, and an analytical pass through the code identifying the specific weaknesses each Capstone enhancement was designed to address.

Weaknesses Identified

Four concrete issues are named in the review and traced back to specific files and lines:

Plaintext Password Storage

The User class stores passwords as a plain string column. The class itself carries the comment, “In real app, hash this! But for class, plain is acceptable.” The UserDao compounds the issue with a literal WHERE username = :username AND password = :password query against that plaintext column. Targeted by Enhancement Three.

allowMainThreadQueries() Anti-pattern

AppDatabase is built with allowMainThreadQueries() enabled, which suppresses Room’s safeguard against accidental UI-thread database access. The flag was used to make the original code easier to write but allows real concurrency bugs to ship undetected. Targeted by Enhancement One.

Broken Delete Path in InventoryAdapter

The delete button removes the item from the adapter’s in-memory list and calls notifyItemRemoved on the RecyclerView, making the card disappear from screen, but never actually invokes the DAO. The deletion only looks like it worked. Targeted by Enhancement One.

Linear Scans in the Read Path

Every read goes through getAllItems() or getZeroStockItems(), each of which scans the full inventory table. The application has no search functionality at all. As inventory grows, this becomes a real performance ceiling. Targeted by Enhancement Two.

Engineering Proposals

The review closes with a walkthrough of the three planned enhancements as engineering proposals rather than as a list of changes. Each proposal names the Course Outcomes it will demonstrate and the trade-offs it accepts. The proposals were deliberately scoped so that all three could be built on top of a single artifact, allowing a reviewer to read the same project evolve from a monolithic Activity into a tested, indexed, server-mediated application.
