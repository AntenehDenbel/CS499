Enhancement 3 · Module Five

Databases and Security

A server-mediated architecture with PBKDF2 password hashing, role-based access control, and an append-only audit log defended by multiple layers.

📂 View enhanced code 📂 View previous milestone
Outcomes 3, 4, 5 Java + Kotlin PBKDF2-HMAC-SHA-256 RBAC · Audit Log 40 Unit Tests

The narrative below is the full Milestone Four submission. It explains the security weaknesses targeted, the components that landed, the substantial deviation from the Module One plan and the reasoning behind each substitution (Firestore became in-process server, BCrypt became PBKDF2, Firestore rules became layered defense), the three course outcomes addressed, and three challenges encountered during implementation.

Artifact Background

The artifact is the Warehouse Inventory Management Android application I originally built in CS 360. It is a Java and Kotlin app that allows a logged in user to track items in a warehouse, see which items are out of stock, add or remove items, and adjust quantities. Through the previous milestones I rebuilt it under an MVVM and Repository structure during Milestone Two and added a custom search index built on a Trie and a min heap during Milestone Three so that lookups against thousands of items run in milliseconds.

For this milestone the database layer itself was the focus. The original artifact stored user passwords in plain text inside the SQLite users table, granted every authenticated user identical privileges, and kept no record of who performed which actions. The Module Five enhancement rebuilds the data and security layers around a server mediated architecture with hashed passwords, role based access control, and an append only audit log, while leaving the UI and search index from the previous milestones untouched.

Why This Artifact

I selected this artifact for the third enhancement because the database and security work here carries the most evidence of growth. The original application would have failed any halfway careful security review, so there was a real ceiling to lift rather than a polish job. The work also aligned with the course outcomes I felt weakest on going into the capstone, particularly the security mindset outcome.

Specifically, the enhancement demonstrates several capabilities I wanted on the record. It shows that I can take an interface boundary I established in an earlier milestone, namely the Repository interface from Milestone Two, and swap the implementation behind it without breaking the UI or the ViewModel layer. It shows that I can model a security architecture around a real threat surface rather than a single feature. It also shows that I can hold the line on those security properties under test, which the new unit test suite verifies.

What I Built

The components that landed are a Kotlin User entity that no longer holds a plaintext password, a PasswordHasher object that runs PBKDF2 with HMAC SHA 256 over a per user salt at the OWASP recommended iteration count, a Session abstraction that holds the currently authenticated user, an InventoryServer class that plays the role a network backend would play and gates every mutation on authentication and role, an AuditEntry table and DAO that records every change with actor, timestamp, and before and after snapshots, and a RemoteInventoryRepository that adapts the existing Repository interface to the new server. Forty new unit tests cover the security properties end to end.

Deviation from the Module One Plan

The Module One plan named Firebase Firestore as the destination for the database migration, BCrypt as the password hasher, and Firestore security rules as the mechanism for enforcing the append only property of the audit log. None of those three choices landed in the final implementation. I want to be honest about why, because the changes were not the result of the plan being wrong but the result of new information surfacing once I started building.

Substitution One: Firestore Became In-Process Server

I was genuinely eager to use Firebase. I had been using it in a personal project of mine and the developer experience was good, the realtime listener model was a clean fit for the LiveData observers I already had in the app, and the offline first behavior was largely free out of the box. When I started writing the code for this milestone I ran into two practical problems. The first is that asking a professor to grade an application that requires a Firebase project, an API key, and a service account JSON introduces a real problem into the grading process. The grader would need to either configure their own Firebase project or trust mine, and neither path is good. The second is that Firebase free tier accounts are reclaimed if they sit idle, which means the artifact would not be particularly long lived as a portfolio piece. A year from now a recruiter clicking the link in my portfolio could easily land on a dead application.

I also considered the database options listed in the Module One assignment instructions, specifically MySQL and MongoDB. Both are server databases, meaning they run as separate processes and listen on a network port, and neither runs on an Android device directly. Integrating either one would have required spinning up a real backend service, deploying it somewhere, and rewriting the application to talk to it over HTTPS. That is roughly a week of additional work, and importantly it would not have changed the security architecture I was building. The authentication, authorization, and audit logic would have looked identical to what I shipped. The only thing that would have changed is the transport. Given that the rubric for the Databases category is about demonstrating advanced database concepts rather than mandating a specific database product, I judged that the better use of the time was to deepen the security work rather than to relocate the same security work onto a different storage engine.

I switched to SQLite, with which I had real shipping experience from a previously released product, and rebuilt the security architecture around it. The trade off I accepted is that the database itself no longer enforces the security properties from the wire side. To compensate I built the InventoryServer class to enforce those properties in code, on its own thread, behind an API boundary the rest of the application cannot reach around. The architecture diagram looks the same as it would have looked with Firestore in the role of the server. The transport differs but the security properties do not, and porting it to a real network service later would primarily mean swapping the executor for a Retrofit client.

Substitution Two: BCrypt Became PBKDF2

For password hashing I had planned to use BCrypt. PBKDF2 with HMAC SHA 256 ended up being the better fit because it ships with the Java standard library through SecretKeyFactory, so I did not need to add a third party dependency for password hashing on a school project. PBKDF2 is also the algorithm NIST Special Publication 800 63B recommends for password storage, with the same security properties I cared about. It is salted with a per user 16 byte salt, slow with a configurable iteration count set to 120,000 which is the OWASP floor as of 2023, and verified with a constant time comparison so a timing side channel cannot leak how many leading bytes of the hash an attacker has guessed correctly. I documented all of this in the doc comment on the PasswordHasher class so that anyone reading the code understands the trade off was deliberate.

Substitution Three: Firestore Rules Became a Layered Defense

For the audit log immutability I had planned to rely on the Firestore rule that denies all writes to the auditLog collection. SQLite has no equivalent mechanism, so I used the next best thing. The AuditDao deliberately declares no Update or Delete annotated methods. Nothing in the ViewModel, the repository, or the UI can amend or remove an existing audit row because no API surface exists for it. The audit log can grow but it cannot be edited. To keep that property honest I wrote a reflection based test that scans AuditDao for any method whose name starts with update, delete, or remove, and fails the build if one ever appears. A future change that breaks the property fails the test rather than silently corrupting the audit trail.

Course Outcomes Addressed

Outcome 5: Security Mindset

The course outcome I had circled going into this milestone was Outcome Five, which is about anticipating adversarial exposures and writing code that mitigates them. The shape of this enhancement is in service of that outcome. Plaintext password storage was an exposure. Lack of authorization was an exposure. Lack of attribution was an exposure. The username enumeration defense I built into the login response is an exposure I addressed proactively rather than reactively. Specifically, the server returns the same generic InvalidCredentials response whether the username was unknown or the password was wrong, which means an attacker cannot probe for valid usernames by watching which response comes back. The audit log internally distinguishes the two cases for an admin reviewing the history later, but the client never sees the difference. Failed login attempts are written to the audit log as LOGIN_FAIL rows specifically so an admin can detect brute force activity in the data.

Outcome 3: Algorithmic Trade-offs

Outcome Three, which concerns designing and evaluating computing solutions using algorithmic principles and trade offs, also applies. Several decisions in this enhancement were trade offs with no clean right answer, and I had to pick one and own the consequences. PBKDF2 instead of BCrypt is one. In process server instead of a cloud database is another. Package visibility instead of database side rules for audit log immutability is a third. In each case I have written the trade off into the code comments and into this narrative rather than letting it pass without explanation.

Outcome 4: Well-Founded Techniques and Tools

Outcome Four, which addresses using well founded techniques to deliver value, shows up in the choice not to break the existing Repository interface. The whole UI and ViewModel layer from Milestones Two and Three runs unchanged on top of the new RemoteInventoryRepository because I held the interface stable. That is the payoff of putting the repository behind an interface in the first place. The data source can change underneath without touching the UI. I see this kind of stable boundary as the difference between code that is rewritable and code that is rigid.

I do not believe I changed my course outcome alignment from the Module One plan. The outcomes I expected this enhancement to demonstrate are the ones it actually demonstrates.

Reflections and Challenges

Challenge One: Java and Kotlin Interoperability

The biggest challenge during the build was the boundary between Java and Kotlin. The existing artifact is mostly Java and the new security and server code is Kotlin. I encountered this problem in two places. The first occurred when InventoryApplication, which is Java, tried to pass System::currentTimeMillis as the clock parameter to InventoryServer, which is Kotlin and expects a function type. Java method references do not auto convert to Kotlin function types. The fix was to add the JvmOverloads annotation to the Kotlin constructor so the Java side could use the production defaults without supplying the clock argument explicitly. The second occurred because Room needs to see annotations on Kotlin entity classes, and the existing build was using annotationProcessor for the Room compiler, which only handles Java sources. I had to switch the build to KSP. Neither of these issues would have appeared in a pure Kotlin or pure Java project. They were specific to the mixed language reality of this artifact, which is the reality found in most working codebases.

Challenge Two: What “Immutable” Means Without Database Rules

The second challenge was thinking carefully about what immutable audit log means in a system that does not have database side rules. Every time I thought I had it locked down I would find another path that could in principle write to it. I ended up with a layered approach. The DAO declares no update or delete methods so there is no API for it. The reflection test enforces that property at build time. The server is the only writer and it never offers an editing method. None of those three layers is sufficient on its own, but together they make the property hard to break by accident, which is the realistic security goal for a single device application.

Challenge Three: LiveData Without a Main Looper

The third challenge involved the Kotlin tests calling into the Java DAOs through fake implementations. The fakes use MutableLiveData to publish updates, which means the choice between setValue and postValue matters depending on whether a main looper is available. JVM unit tests do not have a main looper. The tests would have failed in confusing ways if I had used postValue, so the fakes use setValue, and the tests install InstantTaskExecutorRule to make setValue work synchronously off the main thread. This is the kind of detail that is only learned by running into it.
