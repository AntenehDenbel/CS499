Algorithms and Data Structures

The narrative below is the full Milestone Three submission. It includes the Big-O complexity summary, the benchmark methodology, the actual measured results at three input sizes, an interpretation of the numbers, the course outcomes addressed, an honest correction to the Module One plan, and reflections on two specific bugs that proved most instructive.

Artifact Background

The artifact for this enhancement is the Warehouse Inventory Management Android application I originally built as the Project Three final deliverable in CS 360 (Mobile Architecture and Programming) and re-architected for Milestone Two of CS 499. The starting code for this milestone is the Milestone Two version. It is an Android app written primarily in Java, with Kotlin used for the ViewModels and the Compose theme files, that uses an MVVM architecture with a Repository layer behind an interface, a Room database with both synchronous and LiveData-backed DAO methods, and a small custom Application-scoped service container in place of a full Hilt dependency-injection graph. The user-facing functionality is unchanged from the CS 360 original. Account creation, login, a two-column RecyclerView grid of inventory cards, a bottom sheet for adding items, a second bottom sheet for adjusting quantity with plus and minus buttons, and a low-stock alert path that fires an SMS (or a toast on the emulator) when any item reaches a quantity of zero.

The version I am submitting for Milestone Three keeps every one of those user-facing features and adds one new one, a search box in the toolbar, while replacing the underlying read paths with three coordinated in-memory data structures. The original code answered every read by scanning a LiveData list linearly. The enhanced version answers searches in O(k + m), id lookups in O(1), and low-stock writes in O(log n), by maintaining a Trie, a HashMap, and a binary min-heap behind a single coordinator class called ItemIndex.

Why This Artifact

I chose this artifact for the algorithms and data structures category because the original Milestone Two version had a real performance ceiling that a graduating computer science student should be able to recognize and fix. Continuing to use the same project across all three Capstone categories also means a reviewer can see one artifact evolve from a monolithic Activity in CS 360 into an indexed, tested, MVVM application by the end of CS 499.

The original CS 360 InventoryDao exposed only two queries, getAllItems() and getZeroStockItems(). The first returned every row as a LiveData list. The second returned a full re-scan of the inventory table every time anything changed. There was no search functionality at all. To find a specific SKU, the user had to scroll the grid visually. As long as the warehouse had ten items this was invisible. At a hundred items it was annoying. At a thousand items it would be a blocker. That same linear-scan pattern was also what drove the SMS alert pipeline. Every write triggered a full re-read of the zero-stock subset.

What I Built

The enhancement adds a new package, com.example.inventoryapp.index, containing three classes:

Trie

A hand-written prefix tree keyed on item names. Each node stores the set of items whose names pass through it, which lets prefix searches return matches in O(k + m) time, where k is the length of the search term and m is the number of matches. This is what makes the new SearchView in the toolbar respond character-by-character without ever scanning the full inventory.

MinHeap

A hand-written binary min-heap of items keyed on quantity ascending. The heap maintains an auxiliary HashMap from item id to current heap index, which lets siftUpdate on an arbitrary item run in O(log n) instead of O(n). This is the structure that replaces the original WHERE quantity less than or equal to 0 full-table re-scan. Java already ships PriorityQueue, but its remove(Object) is O(n) because it scans the heap array to locate the element. We needed O(log n) update by id, so the position-map design is justified.

ItemIndex

The coordinator class. It owns the Trie, the HashMap (for O(1) lookup by primary key), and the MinHeap, and it provides a single API surface (search, findById, lowStockTop, upsert, remove, rebuildFrom) that the ViewModel calls. The hard part of maintaining redundant indices is not building them, it is keeping them consistent under mutation. Every write path in ItemIndex is implemented so that the three structures all reflect the same post-state, or none of them do.

The InventoryViewModel was updated to put an ItemIndex between the repository and the UI. The ViewModel now exposes visibleItems, a MediatorLiveData that combines the repository’s snapshot with the search box’s current text and applies the trie filter when the query is non-empty. The Activity adds a SearchView to the toolbar and forwards every keystroke into viewModel.setSearchQuery, which is what makes the type-ahead filter feel instant.

Skills Demonstrated

The skills I aimed to put on display in this enhancement are:

Selecting the right data structure for each access pattern instead of forcing a single structure to do everything. The three operations the app does most (search, lookup, low-stock) each need a differently shaped index, and a single sorted list cannot serve all three.

Hand-implementing a Trie from first principles, including lazy child-map allocation, case-insensitive normalization, and pruning of empty branches on remove.

Hand-implementing a binary min-heap, including the position map that gives O(log n) update-by-id, which is a non-trivial improvement over Java’s built-in PriorityQueue.

Reasoning about Big-O trade-offs in both directions. The index is faster in time but spends additional memory roughly on the order of the sum of all item-name lengths plus a constant per item. For a few thousand SKUs this is a few tens of kilobytes of heap, which is the right trade on Android.

Solving the cache-invalidation problem. Three redundant indices is a classic source of bugs, for example the trie still matching the old name after a rename. The upsert path in ItemIndex removes the previous version of the item from each structure before re-indexing, and a dedicated unit test verifies it.

Writing JUnit micro-benchmarks that produce reportable wall-clock numbers at multiple input sizes (100, 1,000, and 10,000 items), so the Big-O argument is grounded in real measurements rather than asserted in the abstract. The benchmark file also functions as a correctness test. It asserts that the indexed and linear strategies return equivalent results before timing them.

Complexity Summary

Operation Original (linear) Enhanced (indexed)
Search by name prefix O(n), and no UI for it O(k + m) via Trie
Find by id O(n) full scan O(1) via HashMap
Low-stock top-N read O(n) per mutation (re-query) O(n log n) per dashboard read; O(log n) per mutation
Insert / update O(1) DB + O(n) UI re-render O(log n) heap + O(name length) trie

Benchmark Methodology and Results

The benchmark suite (IndexBenchmarkTest) runs each operation under both strategies for 5,000 warm-up iterations followed by 5,000 timed iterations, at three input sizes (100, 1,000, and 10,000 items). The deterministic seed (42) makes results reproducible across runs. The suite asserts that the indexed and linear strategies return equivalent results before timing them, so a regression in correctness would surface as a test failure rather than a quiet drift in the numbers.

Results from running ./gradlew :app:test --tests "*IndexBenchmarkTest*" --info on my development machine:

n Operation Linear (ns/op) Indexed (ns/op) Speed-up
100 search by prefix 4,842 561 8.6x
1,000 search by prefix 43,760 4,056 10.8x
10,000 search by prefix 553,385 106,361 5.2x
100 find by id 1,493 48 31.1x
1,000 find by id 814 34 23.9x
10,000 find by id 12,455 29 429.5x
100 low-stock top-10 9,367 8,269 1.1x
1,000 low-stock top-10 200,199 168,564 1.2x
10,000 low-stock top-10 1,777,349 1,876,379 0.9x

Three observations are worth pulling out from these numbers.

The find-by-id row is the cleanest win, and it does exactly what the Big-O analysis predicts. The linear cost grows with n (1,493 ns at 100, 12,455 ns at 10,000), while the indexed cost stays flat at roughly 30 to 50 nanoseconds regardless of size. By the 10,000-item case the indexed strategy is more than 400 times faster. This is the case for HashMap.

The search-by-prefix row is more interesting. The speedup is real but smaller than I had hoped, and it actually peaks around n = 1,000 rather than continuing to grow at n = 10,000. The reason is that with a synthetic dataset where every item name starts with “item-”, the prefix “item-0” matches roughly a tenth of the inventory at every size. Trie search is O(k + m), and m, the number of matches, scales with n. So the indexed strategy is not constant-time on this benchmark, it just has a much smaller constant factor than walking the full list and doing a per-element startsWith check. In a realistic warehouse where prefixes are more selective (a user typing “scre” probably matches a handful of items, not a thousand), the speedup would be larger. Either way the indexed strategy is meaningfully faster at every size.

The low-stock row is essentially flat, and at n = 10,000 the indexed version is even slightly slower than the linear sort (1.88 ms vs 1.78 ms). This is also exactly what I expected once I corrected my Module One claim about peekTopN ’s complexity. Both strategies are O(n log n) on the read path, so they end up doing structurally similar amounts of work. The win is on the write path. The original code re-runs the LiveData query (a full table scan) on every mutation. The enhanced code does an O(log n) sift on every mutation, and only pays the sort cost when the dashboard is actually rendered. In normal warehouse use mutations greatly outnumber dashboard reads, so the trade is worth it even though the read-side benchmark does not show it.

Course Outcomes Addressed

In Module One I planned this enhancement to address Course Outcome 3 (algorithmic principles and trade-offs) primarily, with Outcome 4 (well-founded and innovative tools and techniques) as a supporting outcome. I believe I met both.

Outcome 3: Algorithmic Trade-offs

The whole point of this enhancement is choosing data structures deliberately and justifying the resulting trade-offs. The Trie was selected because prefix search is the dominant query a SearchView issues. A HashMap is the standard answer when the access pattern is direct lookup by a primary-key-shaped value. A min-heap is the standard answer when the dominant question is “what are the smallest items right now”. None of these were chosen because they are popular. Each was chosen because it matches a specific access pattern in this specific app, and each is contrasted explicitly with the linear strategy it replaces. The narrative includes both a Big-O complexity table and a benchmarked results table, the JUnit suite includes a randomized stress test that compares the indexed and linear strategies for equivalence, and the documentation in MinHeap explicitly discusses where the implementation was kept simple at the cost of theoretical optimality, and what the upgrade path looks like.

Outcome 4: Well-Founded Techniques and Tools

The enhancement introduces an in-memory client-side index, a technique borrowed from database engines, into a mobile application that originally answered every read with a full table scan. It uses Android Architecture Components correctly. A MediatorLiveData that combines two upstream sources is the idiomatic way to build a derived stream in this stack. And it ships a JUnit benchmark suite that produces reportable numbers, which is a tool the original artifact did not have at all.

Correction to the Module One Plan

There is one small update relative to my Module One plan. I originally claimed the low-stock dashboard would read in O(log n) on the heap. That is true on the write side but not on the read side as I implemented it. My peekTopN is O(n log n) on the heap copy. I have updated the class-level Javadoc, the complexity summary in this narrative, and the benchmark commentary so the analysis matches the code. The decision to keep peekTopN simple is defended on engineering grounds (mutations greatly outnumber reads, so paying on read is the cheaper trade overall), but I want a reviewer to see that I caught the discrepancy and updated the documentation rather than left the original claim in place.

I do not believe this milestone fully addresses Outcomes 1, 2, or 5 on its own. Outcome 1 (collaborative environments) and Outcome 2 (professional-quality communication) are continuing themes throughout the ePortfolio, supported here by the new package’s documentation and by this narrative. Outcome 5 (security mindset) is the explicit focus of the Module Five enhancement, where I will replace plaintext passwords with BCrypt hashes, add role-based access control via Firestore security rules, and introduce an immutable audit log.

Reflections

Two things stood out while I was building this enhancement, each of which taught me something I expect to use again.

Reflection One: The Trie Removal Path

The first was that getting the Trie’s removal path right was harder than getting the insertion path right. Insertion is mechanical. Walk the key, allocate nodes lazily, add the item to each node’s set. Removal has to do all of that in reverse and then prune empty branches on the way back up so the tree does not bloat over time. I wrote the recursive removeAt helper twice. The first version forgot to prune an interior node when its child became empty, which meant the trie’s memory footprint grew monotonically across the lifetime of the app. The fix was to make removeAt return a boolean to its parent indicating whether it could be pruned, and to have the parent both delete the child entry and null out the children map when it became empty. The lesson, that the structure of recursion mirrors the structure of the data, and that returning a small piece of state up the call stack is often cleaner than tracking it in a field, is a generalizable one that I had heard in CS 260 but only really internalized here.

Reflection Two: A Subtle Min-Heap Bug

The second was a subtle correctness bug in the MinHeap that the randomized stress test caught and a handful of small examples did not. My initial siftUpdate did only siftUp, on the assumption that the only realistic update was a quantity decrease, because that is what selling stock looks like. The stress test, which applies random quantity updates in both directions, immediately failed. When a top-of-heap item’s quantity went up, it stayed at the top of the heap because nothing pushed it down. The fix was a single line, call siftDown before siftUp, but the lesson is bigger. It is about the value of a randomized test against a sorted-list reference oracle. A sample-based test would have continued to pass, because most random samples happen to exercise the path that worked. The randomized test is the one that exposes the assumption I did not realize I was making.
